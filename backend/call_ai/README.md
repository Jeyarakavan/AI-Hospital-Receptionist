# Open-Source Call AI Hospital Receptionist (Asterisk + Whisper + Piper)

This module adds a narrow, safety-first AI hospital receptionist to your existing project.

## Scope (V1)

Supported intents only:
- `BOOK_APPOINTMENT`
- `CANCEL_APPOINTMENT`
- `RESCHEDULE_APPOINTMENT`
- `GENERAL_INQUIRY`
- `HUMAN_HANDOFF`
- `EMERGENCY`

Non-goals:
- Twilio
- WhatsApp
- Payments
- Multilingual mode
- Generic free-form chatbot behavior

## Architecture

Inbound SIP call (Asterisk) -> `AGI` (`call_ai.agi_receptionist`) -> `faster-whisper` STT -> deterministic intent + slot extraction -> receptionist state machine -> confirmation gate -> Django intake API (`/api/call-ai/intake/`) -> existing DB table (`call_reception_requests`) -> staff UI/API consumption.

TTS is generated by Piper and streamed back to caller as WAV prompts.

## Why this is accuracy-focused

- Deterministic intent routing with strict intent list
- Slot-filling for required fields
- Confirmation before final save
- Emergency phrase immediate override
- Retry count + automatic human handoff
- Confidence thresholds for STT and intent

## Files

- Runtime: `backend/call_ai/agi_receptionist.py`
- State machine: `backend/call_ai/state_machine.py`
- STT: `backend/call_ai/stt_whisper.py`
- TTS: `backend/call_ai/tts_piper.py`
- Asterisk configs: `backend/call_ai/asterisk/*.conf`
- Backend intake API: `backend/api/call_ai_views.py`

## Prerequisites

- Linux host for Asterisk runtime
- Asterisk 18+ (or FreePBX with custom context)
- Python 3.10+
- `ffmpeg`
- Piper binary + English model

## Backend setup

1. Install backend dependencies.
2. Add env vars from `backend/.env.call_ai.example` into your backend `.env`.
3. Ensure `CALL_AI_SHARED_TOKEN` is set in both backend and AGI runtime environment.
4. Run migrations:

```bash
cd backend
python manage.py migrate
```

5. Start Django backend:

```bash
python manage.py runserver 0.0.0.0:8000
```

## Asterisk setup

1. Copy `backend/call_ai/asterisk/extensions_call_ai.conf` to `/etc/asterisk/extensions_call_ai.conf`.
2. Merge `backend/call_ai/asterisk/pjsip_call_ai.conf` sections into your `/etc/asterisk/pjsip.conf`.
3. Ensure your main `extensions.conf` includes:

```conf
#include "/etc/asterisk/extensions_call_ai.conf"
```

4. Place AGI launcher:

```bash
sudo mkdir -p /opt/ai-hospital-receptionist/backend/call_ai
# Deploy repository to /opt/ai-hospital-receptionist
sudo chmod +x /opt/ai-hospital-receptionist/backend/call_ai/agi_entry.sh
```

5. Reload Asterisk:

```bash
asterisk -rx "dialplan reload"
asterisk -rx "pjsip reload"
```

## Run flow

1. Inbound DID routes to `ai-hospital-receptionist` context.
2. AGI answers and plays greeting.
3. Caller speaks; STT transcribes.
4. State machine drives follow-up questions and confirmations.
5. On confirmation, AGI posts structured payload to `/api/call-ai/intake/`.
6. Call ends or transfers based on emergency/handoff behavior.

## Test instructions

Backend tests:

```bash
cd backend
python manage.py test api.tests.test_call_ai_views
```

Manual call tests:

1. Booking path: provide doctor/date/time and confirm with "yes".
2. Cancellation path: provide existing appointment details and confirm.
3. Reschedule path: provide old and new schedule, then confirm.
4. Emergency phrase path: say "chest pain" and verify immediate escalation.
5. Human handoff path: say "operator please" and verify handoff status.
6. Low-confidence path: stay silent/unclear repeatedly and verify `failed_capture`.

## Data saved to backend

Final payload includes:
- `call_id`
- `caller_id`
- `transcript`
- `detected_intent`
- `confidence_score`
- extracted slot fields (patient, doctor, date/time, reason, urgency)
- `confirmation_status`
- `emergency_flag`
- `handoff_flag`
- `final_status`

Records are available through authenticated endpoint:
- `GET /api/call-ai/requests/`
